# PeekDock 设计文档

## 1. 当前结论

PeekDock 的核心不是单独的小屏，也不是单独的桌宠，而是一个跨屏状态迁移体验：

1. Mac 主端承接 agent 的闲置态和召回态，让用户感到 AI 员工在自己的工作桌面上待命。
2. ESP32 小屏承接 agent 的工作态，让用户把 AI 等待从主屏剥离出去。
3. Mac/helper 是两个前端体系之间的状态权威层，负责把 AI 任务状态转换成跨屏事件。

P0 应该优先证明三件事：

1. 任务发起时，agent 能从 Mac 端“进入”小屏，并切到对应工具工作画面。
2. 小屏能横滑查看多个 agent 的运行、完成、需介入、失败状态。
3. 小屏触发完成 review、retry 或澄清后，Mac 端能被召回并打开对应上下文。

当前仓库里仍有早期 PlatformIO/Arduino 串口测试原型，但目标固件应迁移到 ESP-IDF 5.2+ 和 LVGL 9.x。第一阶段需要同时规划 Mac/helper 原型和 ESP32 小屏固件，但小屏端仍是当前 repo 的主要落地点。

本轮决策：

- 黑客松 P0 使用 mock 任务源，不强依赖真实 Codex、Claude、即梦接入。
- 演示 agent 固定为 Codex、Claude 网页端、即梦。
- Mac 端使用 Web 页面模拟主屏 agent 和跨屏动画。
- Mac/helper 与小屏使用 USB Serial，端口为 `/dev/tty.usbmodem1301`。
- 小屏优先使用触摸横滑/上下左右滑动；若触摸手势实现不稳定，降级为屏幕边缘点击切换。
- 快操只打开本地 HTML 或图片结果。
- 目标硬件为 Waveshare ESP32-S3-Touch-LCD-1.47，屏幕为 172 x 320 JD9853 SPI，触摸为 AXS5106L I2C。

## 2. 范围

### P0 范围

- 定义 PeekDock 的任务状态数据模型。
- 定义跨屏互动事件模型，包括从 Mac 到小屏、从小屏到 Mac 的迁移。
- 在 ESP32-S3 固件中渲染小屏工作态。
- 在 Mac 端用轻量原型表现桌面闲置态、发起转场和完成召回。
- 支持 Codex、Claude 网页端、即梦三个 mock 任务源。
- 支持多任务列表和当前任务切换。
- 支持小屏触发 `open_result` 动作，打开本地 HTML 或图片。
- 通过 USB Serial 验证 Mac/helper 与硬件之间的数据收发。

### P1 范围

- Mac 端菜单栏或配置应用。
- AI 工具接入管理。
- Agent 形象管理。
- 更完整动画、跨屏互动、语音输入、随身模式。

### 非目标

- P0 不承诺直接接入所有 AI 工具。
- P0 不做云端账户体系。
- P0 不做复杂任务编排或替用户自动决策。
- P0 不把小屏做成完整副屏，也不承载长文本阅读。

## 3. 核心用户旅程

### 3.1 Mac 端闲置

无任务时，agent 在 Mac 主屏上处于闲置态，例如睡觉、摸鱼、待命或轻微 hover 互动。Mac 端的目标不是持续展示任务详情，而是给用户一个“AI 员工在场”的入口。

### 3.2 任务发起与跨屏进入

用户在 Codex、即梦、Claude 等工具发起任务后，Mac/helper 创建任务并发出 `handoff_to_dock` 事件。Mac 端播放 agent 从桌面离开或向小屏方向移动的动画；小屏端收到事件后播放 agent 出现动画，并切入对应工具的工作画面。

这不是同一个动画对象跨两个屏幕连续移动，而是两端分别播放可对齐的转场段：

- Mac 段：agent 从主屏消失，方向指向小屏。
- 通信段：Mac/helper 发送任务快照和转场事件。
- 小屏段：agent 从边缘或顶部出现，进入工作态。

### 3.3 小屏工作态

小屏显示当前 agent 的工具身份、任务状态、任务短标题、运行时长、可用进度或阶段文案。Codex 可以展示编码/修 bug 工作态，即梦可以展示绘画/喷颜料工作态，Claude 可以展示思考/文档工作态。

### 3.4 小屏多 Agent 横滑

多个任务并行时，小屏以横向卡片或空间切换呈现多 Agent。用户横滑查看不同 AI 员工的空间和状态。P0 优先实现触摸横滑；如果触摸库或手势识别不稳定，降级为屏幕左右边缘点击切换。视觉上仍按横向任务栈设计。

### 3.5 完成与跨屏召回

任务完成后，小屏显示 `Ready to review` 或同类完成态。用户点击/按键后，小屏发送 `open_result` 动作事件。Mac/helper 打开本地 HTML 或图片结果，并触发 Mac 端 agent 召回动画，例如从小屏方向跳回 Mac 主屏。

### 3.6 失败或需要澄清

任务失败或等待用户输入时，小屏显示最小可行动信息和主动作，例如 `Retry`、`Need input`、`Open Codex`。用户触发后，Mac/helper 打开原任务上下文。若是意图澄清，Mac 端可同时开启语音转录或输入框。

### 3.7 趣味联动

P1 可以加入时间和空间联动。深夜状态可以切换为抱咖啡工作；横滑到不同 agent 时，展示对应工作空间，例如 Codex 在 coding room、即梦在云间画图、Browser Agent 在侦探空间。

## 4. 状态模型

### 4.1 任务状态

- `idle`: agent 无任务。
- `queued`: 任务已创建，等待开始。
- `handoff_to_dock`: 任务从 Mac 主端迁移到小屏工作态。
- `running`: 任务进行中。
- `needs_input`: 需要用户补充信息、授权、澄清或处理安全拦截。
- `completed`: 任务已完成，等待 review 或归档。
- `return_to_mac`: 任务完成或需要用户处理，从小屏召回 Mac。
- `failed`: 任务失败，可重试或查看原因。
- `paused`: 用户或系统暂停。

### 4.2 任务字段

- `task_id`: 全局唯一任务 ID。
- `source`: 工具来源，例如 `codex`、`claude`、`jimeng`。
- `agent_name`: 展示名称。
- `title`: 小屏上显示的短任务名。
- `status`: 当前状态。
- `status_text`: 面向用户的短状态文案。
- `started_at`: 任务开始时间。
- `updated_at`: 最近状态更新时间。
- `progress`: 可选进度，范围 0-100。未知时为空。
- `cost_hint`: 可选成本信息，例如 token、积分、生成次数。
- `result_uri`: 可选结果地址或本地路径。
- `actions`: 当前可用动作列表，例如 `open_result`、`retry`、`provide_input`。
- `screen_role`: 当前展示归属，取值为 `mac_idle`、`dock_working`、`mac_recall`。
- `agent_scene`: 小屏空间主题，例如 `coding_room`、`studio_cloud`、`research_room`。
- `animation_key`: 当前应播放的动画资源键，例如 `codex_handoff_in`、`jimeng_running`、`claude_need_input`。

### 4.3 跨屏事件

- `agent_idle_on_mac`: Mac 端显示闲置 agent。
- `task_created`: Mac/helper 创建任务。
- `handoff_to_dock`: Mac 端播放离开动画，小屏端播放进入动画。
- `dock_status_update`: 小屏端更新工作状态。
- `dock_action_event`: 小屏端上报用户动作。
- `return_to_mac`: Mac 端打开结果或上下文，并播放召回动画。
- `sync_snapshot`: Mac/helper 在硬件重启或断连恢复后发送完整任务快照。

## 5. 展示原则

- 小屏只显示“扫一眼就能判断是否要处理”的信息。
- 状态优先于装饰，动画服务于状态识别。
- 未知进度不得显示精确百分比。
- 失败和需要介入必须比完成更高优先级。
- 每个任务的标题需要限制长度，避免小屏文本溢出。
- 快操动作必须有明确结果，不做危险的隐式自动执行。
- 跨屏转场必须允许两端动画略有时间差，以通信确认事件作为状态真相。
- Mac 端负责情绪入口和召回，小屏端负责工作态和任务快操，不把两端职责混在一起。

## 6. 功能行为

### 6.1 小屏任务卡

每个任务卡显示：

- 来源工具和 agent。
- 当前状态。
- 任务标题。
- 运行时长或更新时间。
- 可选进度/成本。
- 当前主动作。

### 6.2 多任务排序

默认排序：

1. `needs_input`
2. `failed`
3. `completed`
4. `running`
5. `queued`
6. `paused`
7. `idle`

### 6.3 快操事件

小屏只发送动作事件，不直接执行本地系统操作。Mac/helper 收到事件后根据白名单执行。

### 6.4 动画状态

P0 可以使用简单帧动画或占位图标。每个状态至少对应一个稳定视觉状态，避免所有状态都只靠文字区分。

### 6.5 Mac 端原型

Mac 端至少包含：

- 桌面或浏览器窗口中的闲置 agent。
- 任务发起按钮或模拟触发器。
- agent 离开 Mac 的动画。
- 接收小屏动作事件后的结果打开/召回动画。

### 6.6 小屏端原型

小屏端至少包含：

- 当前任务工作画面。
- 状态动画位。
- 工具/Agent 标识。
- 横滑或按钮切换多任务。
- 主动作按钮或按键映射。

## 7. 边界情况

- 任务源断连：小屏显示最近状态和断连提示，不清空任务。
- 任务长时间无更新：标记 `stale`，提醒用户可能需要检查。
- 任务结果路径失效：快操失败后提示需要回到 Mac 处理。
- 多任务过多：只显示最高优先级任务和数量，不在小屏塞满列表。
- 硬件重启：从 Mac/helper 重新同步最近任务状态。
- Mac 不在线：小屏进入离线模式，只保留缓存状态。
- 工具不支持进度：显示运行时长和阶段，不显示百分比。

## 8. 验收标准

- 可以通过一条结构化消息创建任务，并在小屏显示来源、标题、状态。
- 任务创建后，Mac 端能播放离开动画，小屏端能播放进入动画。
- 可以更新任务为 `running`、`completed`、`failed`、`needs_input`，小屏视觉状态随之改变。
- Codex、Claude 网页端、即梦三个 mock 任务存在时，可以切换查看，且高优先级任务默认置顶。
- 小屏触发快操后，固件能发出可被 Mac/helper 识别的动作事件。
- 小屏触发 `open_result` 后，Mac 端能打开本地 HTML 或图片结果并播放召回动画。
- 硬件重启后，能重新接收任务快照并恢复显示。
- 未知进度场景不会显示虚假的百分比。

## 9. 仍需用户确认

- Waveshare 官方 ESP-IDF/LVGL demo 中 JD9853 初始化序列和 AXS5106L 寄存器访问代码的具体文件路径。
- 三个 agent 的最终 PNG 帧命名和资产完整度。
- 黑客松是否需要展示真实 Codex/Claude/即梦接入中的任意一个，还是全程 mock 即可。
- 是否坚持可拆卸随身模式进入 P0。
